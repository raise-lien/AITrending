# AITrending 产品 Roadmap

> 基于 2026-07-06 项目探索结果制定  
> 评分方法：RICE（Reach × Impact × Confidence / Effort）

---

## 📊 项目现状速览

| 维度 | 现状 |
|------|------|
| **定位** | AI 信息聚合面板，从 34 个 RSS 源采集展示 AI 领域动态 |
| **技术栈** | Flask + SQLite + feedparser + httpx + APScheduler |
| **架构** | 单文件架构（app.py / fetcher.py / db.py），无构建步骤 |
| **数据量** | 4,701 条条目，34 个源（8 类） |
| **前端** | 原生 HTML/CSS/JS，暗色科技风，单文件 |
| **部署** | 本地运行，端口 5003 |

### 🔍 探索发现的关键问题

| # | 问题 | 严重度 | 影响 |
|---|------|--------|------|
| 1 | **时间戳异常** — 最小值为 `0001-01-01`，存在脏数据 | 🔴 高 | 排序错乱、年份筛选失效 |
| 2 | **937 条 summary 为空**（占 20%） | 🟡 中 | 用户体验下降，搜索覆盖不全 |
| 3 | **Feed name 不一致** — DB 有 "OpenAI Blog" 和 "Hugging Face Blog" 但 feeds.json 已移除/改名 | 🟡 中 | 来源筛选出现幽灵选项 |
| 4 | **无分页机制** — 前端一次加载 limit=3000 全量数据 | 🟡 中 | 数据增长后性能瓶颈 |
| 5 | **无错误处理/重试** — fetcher 失败只 print，无告警 | 🟡 中 | 源静默失效无法感知 |
| 6 | **无用户系统** — 无收藏、无个性化 | 🟢 低 | 单用户工具可接受，但限制增长 |

| 8 | **API 无鉴权** — /api/refresh 任何人可触发 | 🟢 低 | 本地使用可接受，公网部署有风险 |

## 🗺️ Roadmap（按 RICE 优先级排序）

### P0 — 立即修复（数据完整性 & 基础体验）

#### 1. 修复时间戳脏数据
- **问题**：`published_ts` 最小值为 `0001-01-01T00:00:00Z`，导致排序和年份筛选异常
- **方案**：
  - 编写数据清洗脚本，将异常时间戳置为 NULL 或 fetched_at
  - 在 fetcher 中增加时间戳有效性校验（拒绝 1900 年以前的日期）
- **RICE**：R=4701, I=3, C=9, E=1 → **Score: 14103**

#### 2. 修复 Feed Name 不一致
- **问题**：DB 中存在 "OpenAI Blog"（1024条）和 "Hugging Face Blog"（811条），但 feeds.json 中已改为 "OpenAI" 且无 Hugging Face
- **方案**：
  - 数据迁移：将 DB 中的旧 name 映射到 feeds.json 中的新 name
  - 在 fetcher 中增加 feed_name 变更检测机制
- **RICE**：R=4701, I=2, C=9, E=1 → **Score: 8462**

#### 3. 补全缺失的 Summary
- **问题**：937 条（20%）条目无摘要，影响搜索和浏览体验
- **方案**：
  - 排查哪些源不提供 summary（可能是 ArXiv）
  - 对 ArXiv 类源：从 abstract 字段补全
  - 对其他源：在抓取时记录日志，考虑是否移除无内容源
- **RICE**：R=937, I=2, C=8, E=2 → **Score: 3748**

---

### P1 — 近期实现（性能 & 可靠性）

#### 4. 分页加载
- **问题**：前端一次请求 limit=3000 全量数据，数据增长后首屏慢、内存高
- **方案**：
  - 后端：`/api/items` 改为 offset/limit 分页，默认 50 条
  - 前端：滚动到底部自动加载下一页（IntersectionObserver）
  - 保留搜索/筛选状态的一致性
- **RICE**：R=4701, I=3, C=8, E=3 → **Score: 3761**

#### 5. Fetcher 错误处理与监控
- **问题**：fetcher 失败只 print 到 stdout，源失效无法感知
- **方案**：
  - 记录抓取日志到 DB（feed_name, status, error, timestamp）
  - 新增 `/api/health/feeds` 端点：返回各源最近抓取状态
  - 连续 3 次失败的源在前端标记为"异常"
  - 可选：邮件/Webhook 告警
- **RICE**：R=34, I=3, C=8, E=3 → **Score: 272**

#### 6. 搜索增强 — 全文搜索
- **问题**：当前用 SQLite LIKE 搜索，4700+ 条数据性能尚可但会恶化，且不支持中文分词
- **方案**：
  - 启用 SQLite FTS5 虚拟表，建立 title + summary 全文索引
  - 支持中英文混合搜索
  - 搜索结果按相关度排序
- **RICE**：R=4701, I=2, C=8, E=3 → **Score: 2507**

#### 7. 数据库索引优化
- **问题**：当前只有 published_ts 和 feed_name 索引，搜索和分类查询可能全表扫描
- **方案**：
  - 增加 `(feed_name, published_ts)` 复合索引
  - 增加 `guid` 索引（虽然 UNIQUE 自带索引，确认一下）
  - 定期 VACUUM
- **RICE**：R=4701, I=1, C=9, E=1 → **Score: 4231**

## ⚠️ Non-goals（明确不做）

- **不做评论系统** — 信息聚合工具不需要社交属性
- **不做推荐算法** — 基于订阅源的主动选择优于被动投喂
- **不做移动端 App** — 响应式 Web 已够用，不做原生开发
- **不做内容创作** — 只做聚合和展示，不生成原创内容
- **不做付费功能** — 开源工具，保持免费
- **不做多用户系统** — 当前定位为单用户工具
- **不做 Chrome 扩展** — 聚焦 Web 端体验
- **不做邮件/推送订阅** — 暂无需求
